1. Create `client/src/adapter/ws-adapter.ts`:
   - `export class WebSocketAdapter implements EngineAdapter`:
     - Constructor takes `serverUrl: string`, `mode: 'host' | 'join'`, `gameCode?: string`, `deckData: DeckData`
     - Private fields: `ws: WebSocket | null`, `gameState: GameState | null`, `playerId: PlayerId | null`, `playerToken: string | null`, `pendingResolve: ((events: GameEvent[]) => void) | null`
     - `async initialize()`: connect to serverUrl via WebSocket. Set up message handler. If host: send CreateGame with deck. If join: send JoinGame with gameCode and deck. Wait for GameStarted message to resolve.
     - `async submitAction(action: GameAction): Promise<GameEvent[]>`: send Action message, return promise that resolves when StateUpdate received (or rejects on ActionRejected)
     - `async getState(): Promise<GameState>`: return current cached gameState
     - `dispose()`: close WebSocket connection
     - Message handler: parse ServerMessage from each message:
       - GameCreated: store playerToken, gameCode. Wait for opponent. (Emit event or callback for UI to show game code)
       - GameStarted: store state and playerId. Resolve initialize promise.
       - StateUpdate: update cached state, resolve pending submitAction promise with events
       - ActionRejected: reject pending promise
       - OpponentDisconnected: emit reconnect-waiting event (UI shows overlay)
       - OpponentReconnected: dismiss overlay
       - GameOver: update state
     - Expose `gameCode` and `playerId` as readonly getters for UI
     - Store playerToken + gameCode in sessionStorage for reconnection on page refresh

2. Update `client/src/pages/MenuPage.tsx`:

     - "Host Game": navigate to `/game?mode=host`
     - "Join Game": show text input for game code, then navigate to `/game?mode=join&code={gameCode}`
   - All modes accessible from clean card-style layout

3. Update `client/src/pages/GamePage.tsx`:

   - Handle reconnection on page refresh: check sessionStorage for playerToken + gameCode, send Reconnect message

   - `pub struct CardCoverageResult { pub card_name: String, pub set_code: String, pub supported: bool, pub missing_handlers: Vec<String> }`
   - `pub struct CoverageSummary { pub total_cards: usize, pub supported_cards: usize, pub coverage_pct: f64, pub cards: Vec<CardCoverageResult>, pub missing_handler_frequency: Vec<(String, usize)> }`
   - `pub fn analyze_standard_coverage(card_db: &CardDatabase) -> CoverageSummary`:
     - Iterate all cards in the database
     - For each card, check if its set code is in STANDARD_SETS
     - For Standard cards: parse each ability line, check if the ApiType has a registered handler in the effect registry. Check trigger modes against trigger registry. Check keywords against keyword list.
     - Mark card as supported if ALL abilities have registered handlers
     - If any ability's ApiType/trigger/keyword is missing, add to missing_handlers
     - Compute frequency of each missing handler across all unsupported cards
     - Sort missing_handler_frequency descending (most common gaps first)
   - Write unit tests: a card with only supported effects is marked supported; a card with unknown ApiType is marked unsupported with the missing handler listed
