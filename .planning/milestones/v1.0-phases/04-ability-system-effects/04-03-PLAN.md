<task type="auto">
  <name>Task 1: SVar resolution, conditions, and sub-ability chaining</name>
  <files>crates/engine/src/game/effects/mod.rs, crates/engine/src/game/casting.rs</files>
  <action>
**Add to game/effects/mod.rs -- sub-ability chain resolution:**

1. `pub fn resolve_ability_chain(state: &mut GameState, ability: &ResolvedAbility, events: &mut Vec<GameEvent>, registry: &HashMap<String, EffectHandler>, depth: u32) -> Result<(), EffectError>`:
   - If depth > 10, return EffectError::ChainTooDeep (prevents infinite SVar loops)
   - Call `resolve_effect(registry, state, ability, events)` for the current link
   - Check for `SubAbility` key in ability.params:
     - If present, the value is an SVar name (e.g., "DBDraw")
     - Look up the SVar name in `ability.svars` to get the raw ability string
     - Parse with `parse_ability()` (from crate::parser::ability)
     - Build a new ResolvedAbility from the parsed definition, inheriting svars, source_id, controller from the parent
     - Inherit targets from parent if the sub-ability has `Defined$ Targeted` param (the sub-ability operates on the same targets as the parent)
     - Check conditions on the sub-ability link (see condition checking below)
     - If conditions pass, recursively call resolve_ability_chain with depth + 1
   - Also check for `Execute` key (Execute$ pattern) -- same as SubAbility but used for branching

2. **Condition checking** -- `fn check_conditions(ability: &ResolvedAbility, state: &GameState) -> bool`:
   - Check for `ConditionPresent` param: if present, parse the value as a card filter + zone specification. Format examples:
     - `"Card.Self+creaturesYouCtrl"` -- not needed Phase 4, return true
     - For Phase 4, implement basic patterns:
       - Check `ConditionCheckSVar` and `ConditionSVarCompare` params for SVar-based conditions
     - Default: return true (conditions not present = always pass)
   - Check for `ConditionCompare` param: parse comparison string format like `"GE2"` (greater-or-equal 2), `"EQ0"`, `"LE5"`. Compare against a count value from `ConditionSVarCompare` or default to 0. Operators: GE, LE, EQ, NE, GT, LT.
   - Check for `ConditionPresent` param: value is a card property filter. Check if any card matching that filter exists in the specified zone (ConditionZone$ param, default "Battlefield"). For Phase 4, support basic type checks: "Creature" -> any creature on battlefield, "Land" -> any land on battlefield.
   - If no Condition params are present, return true.

3. **Update resolve_top in stack.rs** (or wherever the resolve dispatch happens): Replace the call to `resolve_effect` with `resolve_ability_chain` so sub-abilities automatically chain.

**Update game/casting.rs -- wire card face SVars:**
- When building the ResolvedAbility in handle_cast_spell, populate the `svars` field. The card face SVars need to be accessible. Since GameObjects store `abilities: Vec<String>` but not svars directly, there are two approaches:
  - Option A: Add a `svars: HashMap<String, String>` field to GameObject (populated at object creation time from card data). This is cleaner.
  - Option B: Look up the card definition from a card database reference on GameState.
  - Recommended: Option A -- add `pub svars: HashMap<String, String>` to GameObject (in game_object.rs), defaulting to empty HashMap. When creating objects from parsed cards, the svars will be populated. For test scenarios, populate svars directly on the test GameObject.

Tests in effects/mod.rs:
- resolve_ability_chain with no sub-ability resolves single effect
- resolve_ability_chain with SubAbility$ resolves both effects in sequence
- Chain depth > 10 returns ChainTooDeep error
- ConditionPresent with creature on battlefield returns true
- ConditionPresent with no creatures returns false (sub-ability skipped)
- ConditionCompare GE2 with 3 creatures returns true
- ConditionCompare EQ0 with 1 creature returns false

- Create a 2/3 creature on battlefield controlled by P1 (name="Bear", power=Some(2), toughness=Some(3), card_types with CoreType::Creature)

- P0 passes priority, P1 passes priority -> stack resolves
- Verify: creature has damage_marked == 3, which equals toughness, so SBA should destroy it (moved to graveyard)
- Verify: Lightning Bolt (instant) is in P0's graveyard

**Test 2: Lightning Bolt deals 3 damage to a player**

- Verify: target player's life goes from 20 to 17

- Verify: creature spell is in its owner's graveyard (countered), never reached battlefield
- Verify: Counterspell is in P1's graveyard

- Verify: creature power == 5 (2+3), toughness == 5 (2+3)
