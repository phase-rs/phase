  <behavior>
    - produce_mana(state, object_id, mana_type, player_id) adds a ManaUnit to player's pool with source tracking
    - Tapping a basic land (via ActivateAbility or a dedicated TapLandForMana action) taps the permanent and produces the correct color
    - can_pay(pool, cost) returns true if the pool has enough mana to pay the cost
    - pay_cost(pool, cost) removes the correct mana units: colored shards first, then generic from any available
    - Hybrid shard payment: for W/U hybrid, prefer the color with more available mana in pool
    - Phyrexian shard payment: pay with color if available, otherwise pay 2 life (life payment tracked)
    - X cost payment: X is specified by the player, paid with any mana
    - Generic mana payable by any color or colorless
    - Snow mana: any snow-sourced unit can pay for snow costs
    - ManaPayment action in GameAction for explicit cost payment
    - Mana production generates ManaAdded event
  </behavior>
  <action>
    1. Create `crates/engine/src/game/mana_payment.rs`:
       - `fn produce_mana(state: &mut GameState, source_id: ObjectId, mana_type: ManaType, player_id: PlayerId, events: &mut Vec<GameEvent>)` — creates ManaUnit with source_id, pushes to player's mana_pool, pushes ManaAdded event.
       - `fn can_pay(pool: &ManaPool, cost: &ManaCost) -> bool` — checks if pool can satisfy the cost. For each colored shard, check count_color >= required. For generic, check remaining mana >= generic amount. For hybrid, check if either option satisfiable. Returns bool.
       - `fn pay_cost(pool: &mut ManaPool, cost: &ManaCost) -> Result<Vec<ManaUnit>, PaymentError>` — greedy algorithm: (a) pay colored shards first (exact color match), (b) pay hybrid shards (prefer color with higher count per Forge's approach), (c) pay phyrexian shards (prefer color if available, return LifePayment if not), (d) pay generic from any remaining mana (prefer colorless first, then least-available color). Returns the spent units. If insufficient, return PaymentError.
       - `fn auto_pay_hybrid(pool: &ManaPool, shard: ManaCostShard) -> ManaType` — for a hybrid shard like W/U, returns the color with more available mana in the pool.
       - `pub enum PaymentError { InsufficientMana, InvalidCost }` — derive Debug, Clone, thiserror::Error.

    2. Add `TapLandForMana { object_id: ObjectId }` variant to GameAction in types/actions.rs. This is a convenience action for Phase 3 — tapping a basic land for its mana. In later phases, this generalizes to ActivateAbility.

    3. Handle TapLandForMana in engine.rs apply():
       - Validate: object exists on battlefield, controlled by acting player, is a land, is not tapped.
       - Tap the permanent (set tapped = true).
       - Determine mana color from land's subtypes (Plains->White, Island->Blue, Swamp->Black, Mountain->Red, Forest->Green). Check the GameObject's card_types subtypes.
       - Call produce_mana.
       - Return WaitingFor::Priority { current_player }.

    4. Update game/mod.rs with `pub mod mana_payment;` and re-exports.

    Write tests covering: basic land mana production, can_pay with various costs, pay_cost for colored/generic/hybrid/phyrexian costs, auto-pay preference algorithm for hybrid.

  <behavior>
    - check_state_based_actions loops until no more actions performed (fixpoint), capped at 9 iterations per Forge
    - SBA: player at 0 or less life loses the game (704.5a)
    - SBA: creature with 0 or less toughness is put into graveyard (704.5f)
    - SBA: creature with lethal damage marked is destroyed (704.5g)
    - SBA: legend rule — if player controls 2+ legendaries with same name, choose one to keep, rest go to graveyard (704.5j, auto-choose newest for now)
    - SBA: unattached aura goes to graveyard (704.5n)
    - SBAs generate appropriate events (ZoneChanged for deaths, GameOver for player loss)
    - start_game begins with London mulligan: each player draws 7, then starting player gets MulliganDecision
    - MulliganDecision { keep: true } → if mulligan_count > 0, transition to MulliganBottomCards; if 0, move to next player or start game
    - MulliganDecision { keep: false } → shuffle hand into library, draw 7 again, increment mulligan_count, ask again
    - SelectCards action with card list → puts selected cards on bottom of library (for mulligan bottom-of-library)
    - After both players have kept, start_game calls auto_advance to reach first main phase
    - Full integration test: new_game → mulligan → play land → tap for mana → pass priority through full turn
  </behavior>
  <action>
    1. Create `crates/engine/src/game/sba.rs`:
       - `pub fn check_state_based_actions(state: &mut GameState, events: &mut Vec<GameEvent>)` — fixpoint loop (max 9 iterations per Forge's convention):
         a. `let mut any_performed = false;`
         b. Check 704.5a: for each player, if life <= 0, mark game over. Set any_performed = true, push GameOver event. Set state.waiting_for = WaitingFor::GameOver.
         c. Check 704.5f: for each object on battlefield, if it's a creature (has creature in card_types.core_types) with toughness <= 0, move to owner's graveyard via zones::move_to_zone. Set any_performed.
         d. Check 704.5g: for each creature on battlefield, if damage_marked >= toughness, move to graveyard. Set any_performed.
         e. Check 704.5j: for each player, group legendaries (supertype contains Legendary) by name. If any name has 2+, keep the one with highest entered_battlefield_turn (newest), move rest to graveyard. Set any_performed.
         f. Check 704.5n: for each aura on battlefield, if attached_to is Some but target doesn't exist or isn't on battlefield, move aura to graveyard. Set any_performed.
         g. If !any_performed, break.

    2. Wire SBA into engine.rs: in apply(), after processing any action, call check_state_based_actions(state, &mut events) before auto_advance. If SBAs set waiting_for to GameOver, return that immediately.

    3. Create `crates/engine/src/game/mulligan.rs`:
       - `pub fn start_mulligan(state: &mut GameState, events: &mut Vec<GameEvent>) -> WaitingFor` — shuffle both libraries (using state.rng), draw 7 for each player (move top 7 from library to hand), return WaitingFor::MulliganDecision { player: PlayerId(0), mulligan_count: 0 }.
       - `pub fn handle_mulligan_decision(state: &mut GameState, player: PlayerId, keep: bool, mulligan_count: u8, events: &mut Vec<GameEvent>) -> WaitingFor`:
         - If keep and mulligan_count == 0: move to next player's mulligan (or start game if both done).
         - If keep and mulligan_count > 0: return MulliganBottomCards { player, count: mulligan_count }.
         - If !keep: shuffle hand into library, draw 7 again, return MulliganDecision { player, mulligan_count: mulligan_count + 1 }.
         - If mulligan_count reaches 7: force keep with 0 cards.
       - `pub fn handle_mulligan_bottom(state: &mut GameState, player: PlayerId, cards: Vec<ObjectId>, events: &mut Vec<GameEvent>) -> WaitingFor` — validate cards are in player's hand and count matches expected. Move each card to bottom of library. Then move to next player's mulligan or start game.
       - Internal helper: `fn finish_mulligans(state: &mut GameState, events: &mut Vec<GameEvent>) -> WaitingFor` — both players done. Call auto_advance to reach first main phase.

    4. Wire mulligan into engine.rs:
       - Update start_game to call mulligan::start_mulligan instead of directly auto_advancing.
       - Handle MulliganDecision action: validate waiting_for matches, dispatch to handle_mulligan_decision.
       - Handle SelectCards action when waiting_for is MulliganBottomCards: dispatch to handle_mulligan_bottom.
       - Add SelectCards { cards: Vec<ObjectId> } to GameAction if not already present.

    5. Add events as needed: `PlayerLost { player_id }`, `CreatureDestroyed { object_id }`, `MulliganStarted`, `CardsDrawn { player_id, count }`.

    6. Update game/mod.rs with `pub mod sba;`, `pub mod mulligan;`.

    7. Write a comprehensive integration test in engine.rs or a separate test file:
       - Create a new game with seed.
       - Add basic land cards to each player's library (create GameObjects with appropriate card_types).
       - Start game → get mulligan prompt → both players keep.
       - Play a land from hand → verify it's on battlefield.
       - Tap land for mana → verify mana in pool.
       - Pass priority multiple times → verify phase advances.
       - Complete a full turn cycle → verify turn_number increments.
       - This test proves all Phase 3 systems work together.

    Note: For the integration test, you'll need to create GameObjects that represent basic lands. Since we don't have CardDatabase integration yet, create them directly with appropriate card_types (Land supertype, Plains/Island/etc subtype).
