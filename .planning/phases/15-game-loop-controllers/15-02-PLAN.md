Create `gameLoopController.ts` following the same standalone factory pattern as aiController.ts.

**Interface:**
```typescript
export interface GameLoopConfig {
  mode: "ai" | "online" | "local";
  difficulty?: string;
}

export interface GameLoopController {
  start(): void;
  stop(): void;
  dispose(): void;
}
```

**`createGameLoopController(config: GameLoopConfig): GameLoopController`:**

Internal state:
- `let active = false`
- `let opponentController: OpponentController | null = null` — created for AI mode only
- `let unsubscribe: (() => void) | null = null`

**start():**
1. Set `active = true`
2. If `config.mode === "ai"`, create AI controller via `createAIController({ difficulty: config.difficulty ?? "Medium" })` and call `opponentController.start()`
3. Subscribe to `useGameStore` waitingFor changes via `useGameStore.subscribe(s => s.waitingFor, onWaitingForChanged)`
4. Call `onWaitingForChanged()` once to process current state

**onWaitingForChanged():**
1. If `!active` return
2. Get `waitingFor` from `useGameStore.getState()`
3. If `!waitingFor` or `waitingFor.type === "GameOver"` return
4. If it's the opponent's turn (AI handles this via its own subscription) — return (AI controller's subscription handles AI priority)
5. If it's the player's turn (`waitingFor.data.player === PLAYER_ID` and `waitingFor.type === "Priority"`):
   - Read `fullControl` from `useUiStore.getState()`
   - Read `phaseStops` from `usePreferencesStore.getState()`
   - Call `shouldAutoPass(gameState, waitingFor, phaseStops, fullControl)`
   - If should auto-pass: schedule auto-pass with visual beat delay
6. For non-Priority waitingFor types (MulliganDecision, TargetSelection, etc.) — do nothing, player handles via UI

**scheduleAutoPass():**
- `setTimeout` with `AUTO_PASS_BEAT_MS` (200ms) delay
- Then call `dispatchAction({ type: "PassPriority" })`
- The dispatch function handles animation waiting internally via its mutex/queue
- IMPORTANT: The subscribe fires again after state updates, so the next auto-pass check happens naturally — no explicit loop needed. This avoids the infinite loop pitfall from RESEARCH.md.

**stop():** Set `active = false`, clear any pending setTimeout, stop opponent controller if exists

**dispose():** Call `stop()`, unsubscribe from store, dispose opponent controller if exists, null out refs

**client/src/providers/GameProvider.tsx** — Create React context providing dispatch:

1. Create `GameDispatchContext` via `createContext<(action: GameAction) => Promise<void>>`
   - Default value: a function that throws "No GameProvider" (catches missing provider bugs)

2. `GameProvider` component props: `{ children: React.ReactNode }`
   - This component does NOT own game initialization or adapter creation — that stays in GamePage for now (GamePage knows the mode, deck, etc.)
   - It creates and manages the game loop controller lifecycle
   - It provides `dispatchAction` (from game/dispatch.ts) through context

3. Internal logic:
   - Accept `mode` and `difficulty` as props: `{ mode: "ai" | "online" | "local"; difficulty?: string; children: React.ReactNode }`
   - `useEffect` with `mode` and `difficulty` deps:
     - Create game loop controller via `createGameLoopController({ mode, difficulty })`
     - Call `controller.start()`
     - Return cleanup: `controller.dispose()`
   - Render: `<GameDispatchContext.Provider value={dispatchAction}>{children}</GameDispatchContext.Provider>`

4. Export `useDispatch()` hook: `return useContext(GameDispatchContext)`

**client/src/hooks/useGameDispatch.ts** — Simplify to delegate to standalone dispatch:
- Keep the export signature `useGameDispatch(): (action: GameAction) => Promise<void>` for backward compatibility
- Body: `return useCallback((action: GameAction) => dispatchAction(action), [])`
- Import `dispatchAction` from `../game/dispatch`
- Keep `export { currentSnapshot }` re-exported from `../game/dispatch`
- Remove all the `useRef` mutex logic, `processAction`, `processQueue` — all of that now lives in dispatch.ts
- This ensures existing components using `useGameDispatch()` continue to work unchanged while new code can use `useDispatch()` from context

NOTE: Do NOT remove useGameDispatch entirely — it provides backward compatibility for components not yet migrated to context. The important thing is that both paths go through the same module-level dispatchAction, so there's a single mutex.
