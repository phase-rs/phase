**ETB Tests** (`etb.rs`) -- 3-5 scenarios. Add `use super::*;` at top. Each test uses GameScenario, doc comments reference CR numbers.

Scenarios to implement (adapt to actual GameScenario API from Plan 01 SUMMARY):
1. `/// CR 603.6a: ETB triggered ability fires when creature enters battlefield`
   - Setup: Creature with ChangesZone trigger on battlefield. Cast another creature.
   - Assert: TriggeredAbilityFired event present, or effect of trigger observable (e.g., life gain, damage, draw).
   - Note: If ETB trigger wiring requires specific TriggerMode setup, use `.with_trigger(TriggerMode::ChangesZone)` or whatever the Plan 01 API provides.

2. `/// CR 603.6a: Multiple ETB triggers fire for multiple permanents with triggers`
   - Two creatures with ETB triggers. Third creature enters.
   - Assert: Both triggers fire (check events or game state changes).

3. `/// CR 603.3: ETB triggers go on stack and can be responded to`
   - Step-by-step mode: Creature enters, trigger goes on stack, opponent gets priority before trigger resolves.
   - Assert: Stack has trigger entry, waiting_for is Priority for the appropriate player.

4. `/// CR 614.1c: ETB replacement effects modify how a creature enters`
   - If engine supports ETB replacement (e.g., enters tapped), test that. Otherwise skip this scenario.

**Important:** Work with whatever the engine actually supports. If the engine doesn't have fully wired ETB triggers for arbitrary test cards, write tests using the trigger types that ARE supported (check existing trigger tests in engine.rs and triggers.rs for patterns). The test should exercise real engine paths, not mock behavior.

**Stack Tests** (`stack.rs`) -- 3-5 scenarios:
1. `/// CR 405.5: Stack resolves LIFO -- last spell resolves first`
   - Step-by-step: Cast spell A, cast spell B, resolve. B resolves before A.
   - Assert: Effects of B observed before A (or check resolution order via events).

2. `/// CR 117.3a: Active player gets priority after spell resolves`
   - After a spell resolves, verify WaitingFor::Priority for the active player.

3. `/// CR 405.1: Empty stack after all spells resolve`
   - Cast and resolve all spells. Stack is empty afterward.
   - Assert: state.stack.len() == 0.

4. `/// CR 608.2b: Instant resolves with effect`
   - Cast a bolt (3 damage instant). Resolve it. Target takes 3 damage or loses 3 life.
   - Assert: Life total decreased by 3.

5. `/// CR 117.4: Both players must pass priority in succession for top of stack to resolve`
   - Step-by-step: Cast spell, P0 passes, P1 passes, then top resolves.
   - Verify intermediate state: after P0 passes, P1 has priority.

**Important:** Use the step-by-step `game.act()` mode for stack tests since intermediate state matters. The tests must exercise real engine priority and stack resolution code.

1. `/// CR 704.5a: Creature with lethal damage is destroyed`

   - Assert: Object is in graveyard zone.

2. `/// CR 704.5b: Creature with zero or less toughness is put into graveyard`
   - Creature gets -X/-X reducing toughness to 0 or less.
   - Assert: Object in graveyard.

   - Set player life to 0 or deal lethal damage. SBAs check triggers GameOver.
   - Assert: WaitingFor::GameOver or events indicate loss.

   - Creature takes 1 damage from a source with deathtouch. Any amount of deathtouch damage is lethal.
   - Assert: Object destroyed (in graveyard).

5. `/// CR 704.5k: Indestructible prevents destruction from lethal damage`
   - Indestructible creature takes lethal damage. SBAs do NOT destroy it.
   - Assert: Object remains on battlefield, damage_marked >= toughness but not destroyed.

6. `/// CR 704.5 (general): SBAs checked as a state-based action after each action`
   - Deal damage to creature. Without explicit SBA call, engine should check SBAs automatically.
   - This tests the integration -- engine.apply() runs SBAs as part of the action processing loop.

**Important:** Adapt the scenarios to the actual engine behavior. Check how SBAs are triggered in the existing engine -- likely they're checked automatically after each apply() call. Some scenarios may need combat to deal damage, or direct damage effects. Use the GameScenario builder to set up combat state or use direct damage effects, whichever the engine supports for test cards.

**Targeting Tests** (`targeting.rs`) -- 3-4 scenarios:
1. `/// CR 608.2b: Spell with no legal targets on resolution fizzles`
   - Cast a targeted spell (e.g., bolt targeting a creature). Remove the creature before resolution (e.g., another spell destroys it first). Resolve the bolt.

2. `/// CR 114.1: Spell requires legal targets on cast`
   - Attempt to cast a targeted spell with no valid targets available.

   - Assert: Hexproof creature not in legal targets list, or spell fizzles.

4. `/// CR 114.6: Shroud prevents targeting by any player`
   - Shroud creature can't be targeted even by its controller.
   - Assert: Not in legal targets.

**Important:** The targeting tests depend on how the engine exposes targeting information. Check the WaitingFor::TargetSelection variant -- it includes `legal_targets`. Tests should verify that the engine correctly computes legal targets based on hexproof/shroud, and that resolution checks target legality.
