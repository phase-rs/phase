  <name>Task 1: AnimationOverlay, FloatingNumber, ParticleCanvas components</name>
  <files>client/src/components/animation/AnimationOverlay.tsx, client/src/components/animation/FloatingNumber.tsx, client/src/components/animation/ParticleCanvas.tsx</files>
  <action>
1. Create `client/src/components/animation/FloatingNumber.tsx`:
   - Props: `{ value: number, position: { x: number, y: number }, color: string, onComplete: () => void }`.
   - Renders a number at the given position that floats upward and fades out over ~1 second.
   - Framer Motion: initial={{ opacity: 1, y: 0 }}, animate={{ opacity: 0, y: -60 }}, transition={{ duration: 0.8 }}.
   - Styled: large bold text with text-shadow for readability. Red for damage, green for life gain.
   - Fixed position, pointer-events: none.
   - Calls onComplete when animation finishes.

2. Create `client/src/components/animation/ParticleCanvas.tsx`:
   - HTML5 Canvas overlay for particle effects.
   - Provides imperative API via ref: `emitBurst(x, y, color, count)` for combat strikes, `emitTrail(from, to, color)` for projectiles.
   - Simple particle system: particles have position, velocity, alpha, lifetime.
   - requestAnimationFrame loop renders particles, removes when alpha <= 0.
   - Canvas sized to viewport, position fixed, pointer-events: none, z-index above board but below modals.
   - Particle colors: red for damage, white for combat strikes, cyan for spell effects.

3. Create `client/src/components/animation/AnimationOverlay.tsx`:
   - Reads animation queue from animationStore.
   - When queue is non-empty and isPlaying:
     - Processes effects one at a time (or batches simultaneous ones).
     - For each AnimationEffect, renders the appropriate visual:
       - "damage" -> FloatingNumber at target position
       - "combat_strike" -> ParticleCanvas.emitBurst at defender position
       - "zone_transition" -> (handled by Framer Motion layoutId on cards, so this is mostly a delay for the layout animation to play)
       - "life_change" -> FloatingNumber at life total position
       - "spell_cast" -> brief glow burst on the source card
     - After each effect completes (duration elapsed), calls animationStore.playNext().
   - When queue drains: signals completion so game state renders final state.
   - Wraps FloatingNumber instances in AnimatePresence for clean lifecycle.
  </action>

<task type="auto">
  <name>Task 2: useGameDispatch hook and animation queue integration</name>
  <files>client/src/hooks/useGameDispatch.ts, client/src/stores/animationStore.ts, client/src/pages/GamePage.tsx</files>
  <action>
1. Enhance `client/src/stores/animationStore.ts` (created in Plan 01, expand here):
   - Add `mapEventsToEffects(events: GameEvent[]): AnimationEffect[]` function:
     - DamageDealt -> { type: 'damage', data: { targetId, amount }, duration: 800 }
     - LifeChanged -> { type: 'life_change', data: { playerId, amount }, duration: 600 }
     - AttackersDeclared -> { type: 'combat_strike', data: { attackerIds }, duration: 500 }
     - CreatureDestroyed -> { type: 'creature_death', data: { objectId }, duration: 400 }
     - ZoneChanged -> { type: 'zone_transition', data: { objectId, from, to }, duration: 300 }
     - SpellCast -> { type: 'spell_cast', data: { cardId }, duration: 400 }
     - Other events: no animation effect (skip).
   - `enqueueEffects` pushes mapped effects into the queue and sets isPlaying=true.
   - `playNext` removes first effect from queue; if queue empty, sets isPlaying=false.
   - `positionRegistry`: `registerPosition(objectId: number, rect: DOMRect)` and `getPosition(objectId: number): DOMRect | undefined`. Components call registerPosition in useEffect/useLayoutEffect to keep positions current.

2. Create `client/src/hooks/useGameDispatch.ts`:
   - `useGameDispatch(): (action: GameAction) => Promise<void>`
   - Wraps gameStore.dispatch:
     1. Snapshot current positions (animationStore.positionRegistry).
     2. Call gameStore.dispatch(action) -> returns GameEvent[].
     3. Map events to animation effects via animationStore.mapEventsToEffects.
     4. Enqueue effects via animationStore.enqueueEffects.
     5. Wait for animation queue to drain (subscribe to isPlaying becoming false).
     6. State is already updated (gameStore.dispatch set it), so UI re-renders with final state after animations complete.
   - This is the primary dispatch mechanism -- all game actions should go through useGameDispatch instead of directly calling gameStore.dispatch for proper animation integration.

3. Update `client/src/pages/GamePage.tsx`:
   - Add AnimationOverlay and ParticleCanvas to the page (rendered above the board, below modals).
   - Update all action dispatch calls (PassButton, PlayerHand, targeting, mana payment, modals) to use useGameDispatch instead of direct gameStore.dispatch.
  </action>
