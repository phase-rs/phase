<task type="auto">
  <name>Task 1: Write migration binary</name>
  <files>crates/engine/src/bin/migrate_abilities.rs</files>
  <action>
Create `crates/engine/src/bin/migrate_abilities.rs` — a binary that reads old-format JSON ability files and writes new-format JSON.

Add to engine's `Cargo.toml`:
```toml
[[bin]]
name = "migrate-abilities"
path = "src/bin/migrate_abilities.rs"
required-features = ["forge-compat"]
```

The migration binary must handle these transformations:

**1. TriggerDefinition params -> typed fields:**
Read the old format `{ "mode": "ChangesZone", "params": { "Origin": "Battlefield", "Destination": "Graveyard", "ValidCard": "Card.Self", "Execute": "TrigChange", "TriggerDescription": "..." } }` and produce:
```json
{
  "mode": "ChangesZone",
  "origin": "Battlefield",
  "destination": "Graveyard",
  "valid_card": { "type": "SelfRef" },
  "execute": { ... resolved ability ... },
  "description": "..."
}
```
Map each Forge param key to the typed field per the research mapping table.

**2. StaticDefinition params -> typed fields + modifications:**
Read `{ "mode": "Continuous", "params": { "Affected": "Creature.Other+YouCtrl", "AddPower": "1", "AddToughness": "1", "Description": "..." } }` and produce:
```json
{
  "mode": "Continuous",
  "affected": { "type": "And", ... typed TargetFilter ... },
  "modifications": [
    { "type": "AddPower", "value": 1 },
    { "type": "AddToughness", "value": 1 }
  ],
  "description": "..."
}
```

**3. Forge filter strings -> TargetFilter:**
Parse filter strings like `"Creature.Other+YouCtrl"` into typed TargetFilter JSON. Split on `.` for type and `+` for properties. Map:
- Type part: "Creature" -> TypeFilter::Creature, "Permanent" -> TypeFilter::Permanent, etc.
- "YouCtrl" -> controller: You
- "OppCtrl" -> controller: Opponent
- "Other" -> wrap in And with Not(SelfRef)
- "Self" -> SelfRef
- "tapped" -> properties: [Tapped]
- "EnchantedBy" -> properties: [EnchantedBy]
- Composite: `"Creature.Other+YouCtrl"` -> And([Typed{Creature, You}, Not(SelfRef)])

**4. SubAbility/Execute SVar resolution:**
For each card file, collect all SVars from `remaining_params` entries. When a trigger or ability references "Execute": "SvarName" or "SubAbility": "SvarName", look up the SVar, resolve it into a typed AbilityDefinition, and inline as `execute` or `sub_ability` field.

Track visited SVars with `HashSet<String>` to detect circular references. If a cycle is detected, log the card name and skip (the ability gets `Effect::Unimplemented`).

**5. AbilityDefinition remaining_params -> typed fields:**
- "SpellDescription" / "StackDescription" -> `description`
- "TgtPrompt" -> `target_prompt`
- "SubAbility" -> resolve and inline as `sub_ability`
- "SorcerySpeed" -> `sorcery_speed: true`
- "Duration" -> `duration` (map "UntilEndOfTurn", "Permanent", etc.)
- Drop metadata-only keys: "AILogic", "IsCurse", "Ultimate" (not mechanically relevant)

**6. ReplacementDefinition params -> typed fields:**
- "Execute" -> resolve and inline as `execute`
- "ValidCard" -> parse filter to `valid_card: TargetFilter`
- "Description" -> `description`

**7. AbilityCost Mana string -> ManaCost:**
- Parse mana cost strings like `"1 W"` using `parse_mtgjson_mana_cost()`

**8. Effect variant cleanup:**
- `Effect::Other { api_type, params }` -> `Effect::Unimplemented { name: api_type, description: None }` for unrecognized effects
- `Effect::GenericEffect { params }` -> parse params into `GenericEffect { static_abilities, duration }`
- `Effect::Cleanup { params }` -> map params keys to typed bool fields
- `Effect::Mana { produced: String, params }` -> parse produced string into `Vec<ManaColor>`
- All TargetSpec values -> TargetFilter
- All zone strings -> Zone enum values
- All color strings -> ManaColor enum values

**Execution flow:**
1. Read each `.json` file in data/abilities/
2. Parse as old-format JSON (using serde_json::Value for flexibility)
3. Transform each definition
4. Serialize back as new-format JSON (pretty-printed)
5. Write back to the same file path
6. Print progress (total files, successful, warnings, errors)

Use indicatif for progress bar if already in deps, otherwise simple progress print every 1000 files.

**Round-trip validation**: After writing the new file, re-read it and deserialize as the new typed `AbilityFile` struct to confirm it's valid. Log any deserialization errors.

  </verify>
  <done>Migration binary compiles and can process ability JSON files. Handles trigger params, static params, replacement params, SubAbility resolution, filter string conversion, cost typing, and Effect variant cleanup.</done>
</task>

<task type="auto">
  <name>Task 2: Execute migration and update json_loader</name>
  <files>crates/engine/src/database/json_loader.rs, crates/engine/src/schema/mod.rs, data/abilities/*.json</files>
  <action>
**Step 1: Run the migration binary on all 32K files:**
```bash
cargo run --bin migrate-abilities --features forge-compat -- data/
```
Log output should show total processed, successful, warnings (unresolvable SVars), and errors (parse failures).

**Step 2: Update json_loader.rs** to work with the new typed format:
- CardFace construction: no more svars. Keywords from ability file are already `Vec<Keyword>` (or if the file stores keyword strings, parse them). Power/toughness from MTGJSON are strings — parse into PtValue::Fixed(n) or PtValue::Variable(s).
- The `build_card_face` function constructs a CardFace. Update it:
  - `power: mtgjson.power.as_deref().map(parse_pt_value)` where `parse_pt_value` returns `PtValue::Fixed(n)` for numeric strings or `PtValue::Variable(s)` for "*", "X", etc.
  - `toughness`: same
  - `keywords`: parse MTGJSON keyword strings through `Keyword::from_str()` to get `Vec<Keyword>`
  - Remove svars from CardFace construction (field deleted)
- Implicit ability synthesis (basic lands, equipment, planeswalkers) already constructs AbilityDefinition — update these to use the new field names (no remaining_params).

**Step 3: Update schema/mod.rs:**
- The schema generation uses `schemars::schema_for!(AbilityFile)` which auto-derives from the struct. Since the structs are now typed, the schema will automatically reflect the new types.
- Run the schema generation test to regenerate schema.json: `cargo test -p engine schema::tests`
- Commit the updated schema.json to data/abilities/schema.json

**Step 4: Verify round-trip:**
- Run `cargo test -p engine database::json_loader::tests` to verify the loader works with the new format
- Run `cargo test -p engine schema::tests` to verify schema generation
- Spot-check a few migrated files (lightning_bolt.json, benalish_marshal.json) to verify correct typed format

**Step 5: Rebuild card-data.json:**
- If card-data-export binary exists and is used: `cargo run --bin card-data-export --features forge-compat -- data/`
- Or if the test suite regenerates it, run the relevant test.
