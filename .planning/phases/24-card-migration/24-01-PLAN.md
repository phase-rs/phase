<task type="auto">
  <name>Task 1: Enhance cost parser in ability.rs</name>
  <files>crates/engine/src/parser/ability.rs</files>
  <action>
Replace the stub cost parser at line 465-468 of `parser/ability.rs` with a proper `parse_cost()` function that handles Forge `Cost$` strings.

The current stub:
```rust
let cost = params.remove("Cost").map(|_cost_str| {
    // Cost parsing is basic for now -- Plan 02 will enrich this
    crate::types::ability::AbilityCost::Tap
});
```

Replace with a call to `parse_cost(&cost_str)` that returns `Option<AbilityCost>`:

1. **Tap**: `"T"` -> `AbilityCost::Tap`
2. **Loyalty**: `"AddCounter<N/LOYALTY>"` -> `AbilityCost::Loyalty { amount: N }`, `"SubCounter<N/LOYALTY>"` -> `AbilityCost::Loyalty { amount: -N }`
3. **Sacrifice**: `"Sac<1/CARDNAME>"` or similar `Sac<...>` -> `AbilityCost::Sacrifice { target: TargetSpec::None }` (target detail in the angle brackets is advisory)
4. **Mana**: bare symbols like `"3"`, `"W"`, `"R"`, `"3 W"` -> `AbilityCost::Mana { cost: "3 W".to_string() }`
5. **Composite**: multiple components space-separated (respecting angle brackets), e.g. `"R Sac<1/CARDNAME>"` -> `AbilityCost::Composite { costs: [Mana{..}, Sacrifice{..}] }`
6. **Untap**: `"Untap"` or `"Q"` -> `AbilityCost::Tap` (reuse Tap variant, the engine untap logic is separate)
7. **Unknown components**: anything not recognized (e.g. `"exert"`, `"paylife<2>"`, `"Discard<0/Hand>"`) -> treat as `AbilityCost::Mana { cost: raw_string }` to preserve data without losing information (matches `Effect::Other` fallback pattern)

Implement a helper `split_cost_components(cost_str: &str) -> Vec<String>` that splits on spaces but respects angle bracket nesting (so `Sac<1/CARDNAME>` stays as one token).

Add `#[cfg(test)]` unit tests for parse_cost covering: tap only, mana cost, loyalty add/sub, sacrifice, composite (mana + tap), and unknown fallback.

Wire the new parse_cost into the existing code: `let cost = params.remove("Cost").and_then(|cost_str| parse_cost(&cost_str));`

  </verify>
  <done>Cost parser handles tap, mana, loyalty, sacrifice, composite, and unknown-fallback patterns. All existing engine tests still pass. New unit tests for parse_cost pass.</done>

<task type="auto">
  <name>Task 2: Build migration tool binary and execute migration</name>
  <files>crates/engine/src/bin/migrate.rs, crates/engine/Cargo.toml</files>
  <action>
1. **Add [[bin]] entry** in `crates/engine/Cargo.toml`:
```toml
[[bin]]
name = "migrate"
path = "src/bin/migrate.rs"
```

2. **Create `crates/engine/src/bin/migrate.rs`** as a standalone binary that:

   a. **Accepts args**: `cargo run --bin migrate [-- <forge_dir> <output_dir>]` with defaults `data/cardsfolder` and `data/abilities`.

   b. **Core loop**: Use `walkdir::WalkDir` to recursively traverse `forge_dir`. For each `.txt` file:
      - Read file contents via `fs::read_to_string`
      - Call `engine::parser::parse_card_file(&content)`
      - Convert `CardRules` to `AbilityFile` using a `card_rules_to_ability_file()` function (see RESEARCH.md Pattern 2 for the conversion logic)
      - For single-face cards: populate top-level abilities/triggers/statics/replacements from the face
      - For multi-face cards (all two-face `CardLayout` variants): populate `faces` array using a `face_to_abilities()` helper, leave top-level vecs empty
      - For `Specialize`: base face + all variant faces in `faces` array
      - Derive output filename from `CardRules::name()` using a `card_name_to_filename()` function (lowercase, replace non-alphanumeric with underscore, collapse consecutive underscores, trim trailing underscores)
      - Serialize to JSON via `serde_json::to_string_pretty()`, write to `output_dir/{name}.json` with trailing newline

   c. **Oracle text heuristic** (advisory, not blocking): After parsing, compare extracted keywords from the card's keyword list against common MTG keywords mentioned in oracle text (if available from `CardFace.oracle_text`). Log warnings for potential mismatches to stderr. This is a heuristic check only.

   d. **Statistics tracking**: Track `MigrationStats { total, converted, errors, warnings }`. For each error or warning, record `(source_path, card_name, message)`.

   e. **Output**: Print summary stats to stdout (total/converted/errors/warnings). Write detailed report to `migration-report.json` in the current directory.

   f. **Schema file**: Do NOT overwrite `data/abilities/schema.json` -- only write card JSON files.

3. **Run the migration**: After the binary compiles, execute `cargo run --bin migrate` to generate all ~32,300 JSON files in `data/abilities/`. Verify the tool completes without panics, check summary stats look reasonable (expect some parse errors for malformed Forge cards, expect many warnings for cards with Effect::Other).

Note: The generated files will overwrite the 8 hand-authored JSON files from Phase 23 per user decision. This is correct -- ensures consistency.
