  <action>
    Refactor `client/src/hooks/useGameDispatch.ts` to implement snapshot-before-dispatch:

    The new flow (per user decision — snapshot-before-dispatch is locked):
    1. **Mutex gate**: If `isAnimatingRef.current` is true, queue the action and return a promise that resolves when it's processed. Use a simple ref-based mutex with a pending queue.
    2. **Capture snapshot**: Call `useAnimationStore.getState().captureSnapshot()` to get all card positions before WASM call
    3. **Call WASM**: Call `adapter.submitAction(action)` to get events (do NOT call gameStore.dispatch — call adapter directly to control when state updates)
    4. **Normalize events**: Call `normalizeEvents(events)` from animation/eventNormalizer.ts
    5. **Play animations**: If steps.length > 0 and speed !== 'instant', enqueue steps in animationStore and await playback completion. Pass snapshot to animation system for position lookups.
    6. **Update state**: After animations complete, call `adapter.getState()` and update gameStore with new state. Reproduce the gameStore.dispatch logic (stateHistory, eventHistory, waitingFor) but with deferred timing.
    7. **Release mutex**: Set isAnimatingRef.current = false, process next queued action if any.

    Key implementation details:
    - Use `useRef` for the mutex flag and pending action queue — NOT Zustand state (to avoid re-renders)
    - Read `animationSpeed` from `usePreferencesStore.getState().animationSpeed` (non-reactive read, no re-renders)
    - Read `SPEED_MULTIPLIERS[speed]` to compute actual durations
    - For "await playback completion": subscribe to animationStore.isPlaying going from true to false, or use a promise that resolves after all step durations have elapsed (simpler approach: sum step durations * speed multiplier, then setTimeout)
    - Store the snapshot in a module-level variable or ref so AnimationOverlay can access it for position lookups during playback
    - The `UNDOABLE_ACTIONS` check from gameStore.dispatch must be preserved — check before submitting and save stateHistory accordingly

    Export `captureSnapshot` result access: expose `currentSnapshot` via a module-level ref or via animationStore so AnimationOverlay (Plan 04) can read snapshot positions during playback.
