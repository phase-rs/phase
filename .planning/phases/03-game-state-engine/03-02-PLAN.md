  <name>Task 1: Turn progression and priority system</name>
  <files>
    crates/engine/src/game/turns.rs

  </files>
  <behavior>
    - advance_phase moves to next phase in order (Untap->Upkeep->Draw->...->Cleanup->Untap of next turn)
    - auto_advance skips through phases needing no player input, stops at main phases and end step
    - Untap step untaps all permanents controlled by active player
    - Draw step draws a card (moves top of library to hand) for active player
    - First player on turn 1 skips draw step
    - Cleanup step: active player discards to hand size 7, clear damage on all creatures
    - Phase transitions clear mana pools for all players
    - start_next_turn increments turn_number, switches active_player, resets per-turn counters
    - handle_priority_pass: increments pass counter; if both players passed consecutively with empty stack, advance phase; if both passed with non-empty stack, resolve top of stack; otherwise give priority to opponent
    - Priority resets to active player after each stack resolution
    - Priority resets to active player at start of each phase
  </behavior>
  <action>
    1. Create `crates/engine/src/game/turns.rs`:
       - `const PHASE_ORDER: [Phase; 12]` — all phases in order. Use this for next-phase lookup.
       - `fn next_phase(phase: Phase) -> Phase` — returns next phase, wrapping Cleanup -> Untap.
       - `fn advance_phase(state: &mut GameState, events: &mut Vec<GameEvent>)` — sets state.phase to next, clears mana pools, pushes PhaseChanged event. If wrapping to Untap, calls start_next_turn.
       - `fn start_next_turn(state: &mut GameState, events: &mut Vec<GameEvent>)` — increment turn_number, swap active_player (toggle between 0 and 1), reset priority_player to active_player, reset lands_played_this_turn, reset has_drawn_this_turn, push TurnStarted event.
       - `fn execute_untap(state: &mut GameState, events: &mut Vec<GameEvent>)` — untap all permanents on battlefield controlled by active player (set tapped=false).
       - `fn execute_draw(state: &mut GameState, events: &mut Vec<GameEvent>)` — if library not empty, move top card from library to hand using zones::move_to_zone. Mark has_drawn_this_turn.
       - `fn execute_cleanup(state: &mut GameState, events: &mut Vec<GameEvent>)` — discard down to 7 (for now, auto-discard last cards — proper choice is Phase 7 UI concern). Clear damage_marked on all battlefield creatures.
       - `fn auto_advance(state: &mut GameState, events: &mut Vec<GameEvent>) -> WaitingFor` — loop: match current phase. Untap: execute + advance. Upkeep: advance (no triggers in Phase 3). Draw: execute (skip if turn 1 player 0) + advance. PreCombatMain/PostCombatMain: return Priority. Combat phases: advance (skip in Phase 3). End: return Priority. Cleanup: execute + advance (which starts next turn via start_next_turn, then continues loop).
       - `fn should_skip_draw(state: &GameState) -> bool` — true if turn_number == 1 (first turn of first player).

    2. Create `crates/engine/src/game/priority.rs`:
       - `fn handle_priority_pass(state: &mut GameState, events: &mut Vec<GameEvent>) -> WaitingFor` — increment priority_pass_count. If count reaches 2 (both passed):
         - If stack is empty: advance_phase + auto_advance (return result).
         - If stack is non-empty: resolve top of stack (call stack::resolve_top), reset priority_pass_count, reset priority to active_player, return Priority { active_player }.
       - If count < 2: switch priority_player to opponent, push PriorityPassed event, return Priority { opponent }.
       - `fn reset_priority(state: &mut GameState)` — set priority_player to active_player, set priority_pass_count to 0.
       - `fn opponent(player: PlayerId, state: &GameState) -> PlayerId` — return the other player.

    3. Update `crates/engine/src/game/mod.rs` — add `pub mod turns;` and `pub mod priority;`.

    Note: GameAction::PassPriority may need new variants added for future phases. For now, existing PassPriority variant is sufficient. May need to add PlayLand handling to GameAction in plan 03 (mana payment).

    Note on auto-advance: The research suggests combat phases auto-skip in Phase 3 since there are no creatures attacking. This is correct — combat is Phase 5 scope.
  </action>

    - Mana pools clear on phase transitions
    - Both players passing priority with empty stack advances phase

  <name>Task 2: Stack resolution and apply() engine entry point</name>
  <files>

  <behavior>
    - push_to_stack adds a StackEntry to state.stack (top = end of Vec, LIFO)
    - resolve_top removes last entry from stack, resolves its effect (in Phase 3: spell resolution moves card from stack zone to battlefield or graveyard depending on type)
    - apply(state, action) validates action matches current WaitingFor, dispatches to handler, runs SBAs (stub for now — plan 03), auto-advances, returns ActionResult
    - apply rejects invalid actions (wrong player, wrong phase, etc.) with an error
    - PassPriority action dispatches to priority::handle_priority_pass
    - PlayLand action: validates it's a main phase, player has priority, hasn't exceeded land limit; moves card from hand to battlefield; resets priority pass count
    - EngineError enum for action validation failures
    - apply returns Result<ActionResult, EngineError>
  </behavior>
  <action>
    1. Create `crates/engine/src/game/stack.rs`:
       - `fn push_to_stack(state: &mut GameState, entry: StackEntry, events: &mut Vec<GameEvent>)` — push to state.stack Vec (last = top for LIFO), push event.
       - `fn resolve_top(state: &mut GameState, events: &mut Vec<GameEvent>)` — pop last from state.stack. Match on kind: Spell → for Phase 3, if the card is a permanent type (creature, artifact, enchantment, planeswalker, land — check card_types), move from Stack zone to Battlefield; if instant/sorcery, move to Graveyard. Push a SpellResolved or similar event. Use zones::move_to_zone for the transfer.
       - `fn stack_is_empty(state: &GameState) -> bool`.

    2. Create `crates/engine/src/game/engine.rs`:
       - `pub enum EngineError` with variants: InvalidAction(String), WrongPlayer, NotYourPriority, ActionNotAllowed(String). Derive Debug, Clone, thiserror::Error.
       - `pub fn apply(state: &mut GameState, action: GameAction) -> Result<ActionResult, EngineError>`:
         a. Validate action is legal given current state.waiting_for (e.g., if WaitingFor::Priority { player }, only that player can act).
         b. Match on action:
            - PassPriority → priority::handle_priority_pass
            - PlayLand { card_id } → validate main phase, land limit not exceeded. Find the object in player's hand by matching card_id. Move from hand to battlefield via zones::move_to_zone. Increment lands_played_this_turn. Reset priority_pass_count (action was taken, passes reset). Set tapped=false on the land. Return WaitingFor::Priority { current_player } (player retains priority after playing a land per MTG rules).
            - Other actions → return EngineError::ActionNotAllowed for now (CastSpell handling is partial in Phase 3, full implementation is Phase 4).
         c. Run state-based actions (stub: call sba::check_state_based_actions which will be a no-op until plan 03).
         d. Return ActionResult { events, waiting_for }.
       - `pub fn new_game(seed: u64) -> GameState` — convenience to create a two-player game. Delegates to GameState::new_two_player.
       - `pub fn start_game(state: &mut GameState) -> ActionResult` — begins the game: set turn_number to 1, run auto_advance to get to first main phase (or mulligan if we add that in plan 03). Returns initial ActionResult.

    3. Add new GameEvent variants if needed: `LandPlayed { object_id, player_id }`, `StackResolved { object_id }`. Update events.rs.

    4. Add new GameAction variants if needed: consider adding `SelectCards { cards: Vec<ObjectId> }` for mulligan bottom-of-library selection (used in plan 03).

    5. Update `crates/engine/src/game/mod.rs` — add `pub mod stack;`, `pub mod engine;`, re-export apply, EngineError, new_game, start_game.

    Write integration tests in engine.rs that exercise a mini game loop: create game → start → pass priority through phases → verify phase progression and events.
