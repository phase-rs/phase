  <name>Task 2: Engine integration -- trigger processing after actions</name>

Add to `crates/engine/src/game/triggers.rs`:
- `pub fn process_triggers(state: &mut GameState, events: &[GameEvent])`: the main entry point called by engine.rs after action processing. This function:
  1. Builds trigger registry (same pattern as effect registry -- build per call, cheap)
  2. For each event, scans all permanents on the battlefield for TriggerDefinitions (stored on GameObject -- the executor must check if trigger_definitions field exists on GameObject; if not, parse triggers from the card's ability strings using the existing ability parser, OR add a `trigger_definitions: Vec<TriggerDefinition>` field to GameObject populated at creation time)
  3. For each matching trigger, creates a PendingTrigger with source_id, controller, timestamp
  4. Sorts pending triggers APNAP: `sort_by_key(|t| (if t.controller == state.active_player { 0 } else { 1 }, t.timestamp))`
  5. Reverses the sorted list (so AP triggers end up on top of stack, resolving last per LIFO)
  6. For each pending trigger, creates a ResolvedAbility from the TriggerDefinition (api_type from trigger's "Execute" param or the trigger def's associated ability, params from trigger def, source_id, controller from trigger's source object)
  7. Pushes a StackEntry with StackEntryKind::TriggeredAbility onto state.stack

For triggered ability construction: look at how the trigger's params reference an ability to execute. In Forge, triggers have an "Execute" param pointing to an SVar that defines the ability. The ResolvedAbility for the trigger should be built from this SVar resolution (reuse existing SVar resolution from Phase 04). If no "Execute" param, check for inline ability params (api_type, NumDmg, etc.).

Add `trigger_definitions: Vec<TriggerDefinition>` field to `GameObject` if not already present. Populate it during object creation in `create_object` / casting flow by parsing T: lines from card data. This avoids re-parsing on every trigger scan.

Integrate into `crates/engine/src/game/engine.rs`:
- After the main action processing switch block produces events (before returning ActionResult), call `triggers::process_triggers(state, &events)` to check for and place triggered abilities on the stack.
- This must happen AFTER state-based actions are checked (SBAs may produce additional events like CreatureDestroyed that should also trigger).
- If triggers were placed on the stack, priority should be given to the active player (triggers go on stack, then priority is granted).
- The existing `handle_priority_pass` flow already handles stack resolution, so triggered abilities on the stack will resolve normally through the existing priority/stack system.

Integration flow in apply():
1. Process action -> events
2. Run SBAs (may produce more events)
3. Call process_triggers(state, &all_events)
4. If triggers were placed on stack, return WaitingFor::Priority to let players respond
