1. Update `client/src-tauri/tauri.conf.json`:
   - Add `"bundle": { "externalBin": ["binaries/phase-server"] }` to the bundle config
   - This tells Tauri to bundle the phase-server binary as a sidecar

2. Update Tauri capabilities for shell plugin:
   - Read `client/src-tauri/capabilities/default.json` and add shell permissions:
     ```json
     {
       "permissions": [
         "..existing..",
         "shell:allow-spawn",
         "shell:allow-kill",
         "shell:allow-stdin-write"
       ]
     }
     ```
   - If capabilities file doesn't exist in the expected format, check `client/src-tauri/Cargo.toml` for plugin registration and add `tauri-plugin-shell` if needed.
   - Install the plugin: add `tauri-plugin-shell` to Cargo.toml dependencies if not present.

3. Create `client/src/services/sidecar.ts`:
   - Import `Command` from `@tauri-apps/plugin-shell` (lazy/conditional import since this only works in Tauri)
   - `isTauri(): boolean` — check `window.__TAURI_INTERNALS__` exists
   - `spawnSidecar(port?: number): Promise<{ port: number; kill: () => Promise<void> }>`:
     - Default port: 8080. Try ports 8080-8089 if in use.
     - Create Command.sidecar with env `PORT={port}` and `PHASE_DATA_DIR` pointing to appropriate data directory
     - Spawn the process
     - Health check: poll `http://localhost:{port}/health` every 500ms up to 10 attempts (5s timeout)
     - On health check success, return `{ port, kill }` where kill terminates the child process
     - On failure, kill process and throw
   - `stopSidecar(handle: { kill: () => Promise<void> }): Promise<void>`:
     - Call handle.kill()
   - Export a module-level `sidecarHandle` ref for cleanup

4. NOTE: The actual sidecar binary must be placed at `client/src-tauri/binaries/phase-server-{target-triple}`. This is a build step, not a runtime step. Add a comment in sidecar.ts explaining:
   ```
   // To set up the sidecar binary for development:
   // cargo build -p phase-server
   // cp target/debug/phase-server client/src-tauri/binaries/phase-server-$(rustc --print host-tuple)
   ```

  <action>
1. Implement smart server detection in `GameProvider.tsx` (or a new `client/src/services/serverDetection.ts`):
   - `detectServerUrl(): Promise<string>`:
     - Step 1: If `isTauri()`, try `http://localhost:8080/health`. If responds, use `ws://localhost:8080/ws`
     - Step 2: Read `useMultiplayerStore.getState().serverAddress`. If non-empty, try health check. If responds, use it.
     - Step 3: Fall back to manual entry (return stored address or default)
   - Use this in GameProvider's online mode setup instead of hardcoded `getWsUrl()`

2. Implement CODE@IP:PORT join code parsing:
   - In LobbyView's join-by-code flow, parse the code: if it contains `@`, split into `code` and `address`
   - `parseJoinCode(input: string): { code: string; serverAddress?: string }`:
     - Input: "ABC123" -> `{ code: "ABC123" }`
     - Input: "ABC123@192.168.1.5:8080" -> `{ code: "ABC123", serverAddress: "ws://192.168.1.5:8080/ws" }`
     - Input: "ABC123@myserver.com" -> `{ code: "ABC123", serverAddress: "ws://myserver.com:8080/ws" }` (default port 8080)
   - When serverAddress is present, connect to that server instead of the default

3. Create `client/src/components/multiplayer/ConnectionDot.tsx`:
   - Reads `connectionStatus` from `useMultiplayerStore`
   - Renders a small circle (8px): green for `connected`, yellow/pulsing for `connecting`, red for `disconnected`
   - Fixed position: top-right area, subtle, visible but not intrusive
   - Use framer-motion for gentle pulse animation on `connecting` state
   - Only renders when in multiplayer mode (check if connectionStatus is not disconnected, or always show during online game)

4. Update `multiplayerStore.ts` with connection lifecycle tracking:
   - Ensure `setConnectionStatus` is called at the right times:
     - When WebSocket opens: `'connecting'`
     - When GameStarted received: `'connected'`
     - When WebSocket closes/errors: `'disconnected'`
   - Add these calls in ws-adapter.ts or GameProvider.tsx

5. Connection failure toast:
   - When WebSocket connection fails in GameProvider (the initReject fires), show a toast via a simple notification mechanism.
   - Create a minimal toast: fixed-position div at bottom of screen that auto-dismisses after 5 seconds
   - Toast text: "Connection failed. Retry or change server in Settings."
   - Two action buttons: "Retry" (re-attempt connection) and "Settings" (open preferences modal)
   - Can use a simple state in multiplayerStore: `toastMessage: string | null`, `clearToast()`

6. Wire ConnectionDot into GamePage.tsx:
   - Add `<ConnectionDot />` to the GamePage layout (only in online/p2p mode)
   - Position in top-right area near the game menu
