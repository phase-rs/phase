  <name>Task 1: Promote Ward, Protection, CantBeBlocked static abilities and integrate with targeting/combat</name>
  <files>crates/engine/src/game/static_abilities.rs, crates/engine/src/game/targeting.rs, crates/engine/src/game/combat.rs</files>
  <behavior>
    - Test: Creature with CantBeBlocked static cannot be assigned blockers -> validate_blockers returns Err
    - Test: Creature without CantBeBlocked can be blocked normally
    - Test: Ward check returns true when targeting a creature with Ward (indicating opponent must pay cost)
    - Test: Protection from red prevents red source from targeting (targeting validation fails)
    - Test: Protection from red prevents red creature from blocking (combat validation fails)
    - Test: Protection from red does NOT prevent non-red source from targeting
  </behavior>
  <action>
1. In `static_abilities.rs`:
   - Replace `handle_stub` for "CantBeBlocked" with `handle_cant_be_blocked` that returns `RuleModification { mode: "CantBeBlocked".to_string() }`
   - Replace `handle_stub` for "Ward" with `handle_ward` that returns `RuleModification { mode: format!("Ward:{}", cost) }` where cost comes from params
   - Replace `handle_stub` for "Protection" with `handle_protection` that returns `RuleModification { mode: format!("Protection:{}", target) }` where target comes from params
   - Move these 3 from the `stubs` array to explicit `registry.insert()` calls with their new handlers

2. In `combat.rs` `validate_blockers()`:
   - After the existing evasion checks (Flying, Shadow, Fear, etc.), before the Menace check:
   - Check if attacker has a "CantBeBlocked" static ability active via `check_static_ability(state, "CantBeBlocked", &context)` or by checking if the attacker object has a CantBeBlocked static definition
   - Simpler approach: check `attacker.static_definitions` for a CantBeBlocked mode entry, since this is how Forge cards encode it. If found, return Err("cannot be blocked").
   - Also add Protection blocking restriction: if attacker has Protection from a quality that matches the blocker (e.g., Protection from Red and blocker is red), the blocker cannot block it

3. In `targeting.rs`:
   - Add Ward check: when validating a target, if the target permanent has Ward, the targeting is still legal but should emit an event/flag indicating the Ward cost must be paid. For now, add a comment that Ward cost payment integration will come with the mana payment UI (Phase 17). The important part is that the static is recognized and not a stub.
   - Add Protection targeting restriction: if target has Protection from a quality matching the source (e.g., Protection from Blue and source is blue), targeting is illegal

4. Add tests in combat.rs and targeting.rs `#[cfg(test)]` modules.
  </action>

  </verify>
  <done>CantBeBlocked prevents blocking, Ward is recognized (cost enforcement deferred to Phase 17), Protection restricts targeting and blocking by matching quality</done>
