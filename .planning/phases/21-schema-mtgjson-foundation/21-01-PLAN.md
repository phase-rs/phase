IMPORTANT: This task ONLY defines types. It does NOT update consumers (engine.rs, casting.rs, etc.) -- that is Plan 02. To keep the crate compiling, the Forge parser and all consumers still need to work. The approach: keep the OLD AbilityDefinition, TriggerDefinition, etc. as type aliases or maintain backward compat via the parser returning typed structs. However, since the parser's `parse_ability()` returns `AbilityDefinition` and the consumers read `def.api_type` and `def.params`, the cleanest approach is:

Strategy for compilation: The Forge parser (`parser/ability.rs`) creates AbilityDefinition. Currently it sets `api_type` and `params`. After this task, AbilityDefinition has `effect: Effect` instead. The parser must be updated to produce `Effect` variants from the parsed strings. This is straightforward -- the parser already extracts `api_type` and `params`, so it can map `api_type` to the appropriate `Effect` variant, falling back to `Effect::Other { api_type, params }` for any it doesn't recognize.

Similarly, `parse_trigger()` currently returns `TriggerDefinition { mode: String, params }`. Update to return `TriggerDefinition { mode: TriggerMode::from_str(&mode_str), effect: ..., params }`. The effect field can default to `Effect::Other` since triggers' effects are resolved differently.

`parse_static()` returns `StaticDefinition { mode: String, params }`. Update to `StaticDefinition { mode: StaticMode::from_str(&mode_str), params }`.

`parse_replacement()` returns `ReplacementDefinition { event: String, params }`. Update to `ReplacementDefinition { event: ReplacementEvent::from_str(&event_str), params }`.

Update `parser/ability.rs` to produce the new typed structs. This is part of defining the types -- without it the crate won't compile.

Also update `ResolvedAbility` to use `effect: Effect` instead of `api_type: String, params: HashMap`. This is referenced by the effect resolution system, so `effects/mod.rs` will need the registry changed from `HashMap<String, EffectHandler>` to matching on `Effect` variants. BUT that is a huge consumer change. To keep this plan scoped, use a transitional approach:

TRANSITIONAL APPROACH: Keep `ResolvedAbility` with `api_type: String` and `params: HashMap<String, String>` FOR NOW. Plan 02 will convert it. The reason: ResolvedAbility is used in StackEntry, PendingCast, and by all effect handlers. Changing it here would require updating all 12 consumer files in this same task, which defeats the plan split.

So the deliverable for this task:
1. New typed enums: Effect, TargetSpec, DamageAmount, AbilityCost, StaticMode, ReplacementEvent
2. Updated definition types: AbilityDefinition (typed), TriggerDefinition (typed mode), StaticDefinition (typed mode), ReplacementDefinition (typed event)
3. ResolvedAbility: UNCHANGED (transitional -- Plan 02 converts it)
4. Updated parser/ability.rs to produce typed definition structs
5. All derives including JsonSchema
6. Round-trip serde tests

For compilation: The consumers that read `def.api_type` and `def.params` (casting.rs, coverage.rs, mana_abilities.rs, effects/mod.rs) now need to access `def.effect` instead. Since we're keeping the consumer update for Plan 02, we need a compatibility path. Add helper methods on AbilityDefinition:

This keeps all existing consumers compiling while the types are updated. Plan 02 removes these helpers.
