<objective>
Feature-gate all Forge-specific code behind `forge-compat`, delete Forge data, and adapt the coverage binary and CI to JSON-only operation.

Purpose: After Plan 01 eliminated all compat method calls from production code, the compat methods (api_type, params, mode_str, event_str), the Forge parser, the Forge loader, and the migration binary can be safely gated. The data/standard-cards/ directory (78 tracked .txt files) is no longer needed since JSON is the sole data path. The coverage report binary's text/Forge mode is removed, making JSON the default. CI is updated to a single JSON coverage gate.

Output: The engine compiles without Forge code by default. `cargo test --all` passes. `cargo build --target wasm32-unknown-unknown -p engine-wasm` succeeds. data/standard-cards/ deleted. CI uses JSON-only coverage gate.
</objective>

```

CRITICAL: `parse_ability()` in parser/ability.rs is called by:
1. `game/effects/mod.rs` line 1: `use crate::parser::ability::parse_ability;` -- used in resolve_ability_chain() for SubAbility SVar parsing (PRODUCTION CODE, must stay ungated)
2. `game/coverage.rs` line 10: `use crate::parser::ability::parse_ability;` -- used for SVar coverage checking (PRODUCTION CODE, must stay ungated)

So: Gate `card_parser`, `card_type`, `mana_cost` modules and `parse_card_file` re-export. Keep `ability` module ungated.

Existing imports to handle:

    pub use card_parser::parse_card_file;

    // ParseError stays ungated -- used by ability parser
    ```

    **Step 3: Feature-gate `CardDatabase::load()` in `database/card_db.rs`.**
    Wrap the `load()` method and its `use crate::parser::{parse_card_file, ParseError};` import in `#[cfg(feature = "forge-compat")]`. The `load_json()` method and all other database code stays ungated. The ParseError import specifically for load() should be gated; if ParseError is used elsewhere in card_db.rs, only gate the parse_card_file import.

    **Step 4: Feature-gate compat bridge methods in `types/ability.rs`.**
    Gate these methods with `#[cfg(feature = "forge-compat")]`:
    - `Effect::api_type(&self) -> &str`
    - `AbilityDefinition::api_type(&self) -> &str`
    - `AbilityDefinition::params(&self) -> HashMap<String, String>`
    - `ResolvedAbility::from_raw()` constructor
    - Any `mode_str()` method on StaticDefinition, `event_str()` on ReplacementDefinition

    Keep ungated:
    - `Effect::to_params()` -- legitimate serialization used by casting.rs
    - `effect_variant_name()` standalone function (added in Plan 01)
    - All typed enum definitions, Display impls, FromStr impls

    **Step 5: Gate any test-only compat usage.**
    Search for `api_type()`, `from_raw()`, `mode_str()`, `event_str()`, `params()` calls in `#[cfg(test)]` modules throughout the engine. For each:
    - If the test module is in a file that has both production and test code, wrap the specific test functions using compat methods in `#[cfg(feature = "forge-compat")]`
    - Alternatively, refactor the test to use typed APIs (preferred if simple)
    - `parse_test_ability()` helper functions in test modules that call `parse_ability()` are fine -- `parse_ability` stays ungated
    - Test code in `copy_spell.rs` using `api_type()`: either refactor to `effect_variant_name()` or gate

    **Step 6: Delete `crates/engine/tests/parity.rs`.**
    This file contains integration tests for Forge-vs-JSON parity validation, which has been completed. Delete the file entirely per user decision (not feature-gated, just deleted).

    **Step 7: Verify compilation without forge-compat.**
    Run `cargo build -p engine` (no features). Fix any `unresolved import` errors from gated modules. Common issues:
    - Dead `use crate::parser::parse_card_file` in card_db.rs if not properly gated
    - Dead `use crate::parser::ability::parse_ability` if ability module was accidentally gated
    - Test code calling gated methods

    Run `cargo build -p engine --features forge-compat` to verify Forge code still compiles when feature is enabled.

    Run `cargo fmt --all` after all changes.
