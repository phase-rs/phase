<objective>
Build the JSON card loader that merges MTGJSON metadata + ability JSON files into engine-ready CardFace/CardRules objects.

Purpose: This is the core data pipeline that replaces the Forge .txt card loading path. Every subsequent phase (24: migration, 25: Forge removal) depends on this loader working correctly. The loader must handle single-face cards, multi-face cards, implicit ability synthesis (basic lands, equipment, planeswalkers), and threading of scryfall_oracle_id for frontend image lookups.

Output: `json_loader.rs` module with `load_json()` method on `CardDatabase`, `FaceAbilities` extension to `AbilityFile`, `scryfall_oracle_id` on `CardFace`, updated `parse_loyalty_cost()` in planeswalker.rs. All with unit tests.
</objective>

   **`build_card_type(mtgjson: &AtomicCard) -> CardType`**: Map MTGJSON `supertypes` -> `Supertype::from_str()`, `types` -> `CoreType::from_str()`, `subtypes` -> `Vec<String>`. Skip unrecognized supertypes/core types (they fail `from_str`).

   **`map_layout(layout_str: &str) -> LayoutKind`** (internal enum): Map MTGJSON layout strings to the engine's layout variants. Use match: "normal"|"saga"|"class"|"case"|"leveler" -> Single, "split" -> Split, "flip" -> Flip, "transform" -> Transform, "meld" -> Meld, "adventure" -> Adventure, "modal_dfc" -> Modal, _ -> Single.

Exact pattern from `scenario.rs:153-162`.

   **`synthesize_equip(face: &mut CardFace)`**: Per CR 702.6 and user decision in CONTEXT.md, synthesize an Equip activated ability from the card's keywords. For each keyword string matching "Equip" or "Equip:{cost}" pattern: extract the cost string (e.g., "2" from "Equip:2", or "{1}{W}" from "Equip:{1}{W}"). Push an `AbilityDefinition { kind: Activated, effect: Effect::Other { api_type: "Attach".to_string(), params: HashMap::new() }, cost: Some(AbilityCost::Mana { cost: cost_str }), sub_ability: None, remaining_params: HashMap::new() }`. If the keyword is bare "Equip" with no cost parameter, skip synthesis (cost is unknown). Note: The current engine's `handle_equip_activation()` checks subtypes, not abilities, but synthesizing the ability makes the equip cost explicitly accessible and aligns with CR 702.6 ("Equip [cost]" means the card has an activated ability with that cost). This is forward-correct design for when the engine reads equip costs from abilities.

   **`build_card_rules(mtgjson_faces: &[AtomicCard], ability_file: &AbilityFile, oracle_id: Option<String>) -> CardRules`**: Determine layout via `map_layout()`. For single-face: build one CardFace from flat AbilityFile fields. For multi-face: use `ability_file.faces` (fall back to flat fields for face 0 if faces is empty). Build appropriate `CardLayout` variant. Apply `synthesize_basic_land_mana()` and `synthesize_equip()` to each face. Return `CardRules { layout, meld_with: None, partner_with: None }`.

   **`load_json(mtgjson_path: &Path, abilities_dir: &Path) -> Result<CardDatabase, Box<dyn Error>>`**: (This will be called as a function, then wired into `CardDatabase`.)

   - Build `cards` HashMap and `face_index` HashMap (same pattern as existing `load()`)
   - Collect errors for files that fail to parse or can't find MTGJSON match
   - Return `CardDatabase` with populated cards/face_index/errors
